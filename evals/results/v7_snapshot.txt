================================================================================
  SYSTEM PROMPT
================================================================================

  Role: Analyst assistant for a B2B software company.

  Tools described inline (four tools):

  search(query, ...)
    Full-text search over all artifact content. Returns matching artifacts with
    a content snippet. Use as first move for almost every content question.
    Set group_by_customer=True for pattern or classification questions.
    Filters: customer_name, region, crm_stage, artifact_type.

  read(artifact_ids)
    Returns FULL content for one or more artifacts by ID. Pass multiple IDs in
    a single call.
    ** New in v7 ** ALWAYS call read() when the question asks for specific
    technical values: field names, commands, exact percentages, exact dates,
    action-item owners, step-by-step plans. Snippets are teasers — never try
    to answer a detail question from a snippet alone.

  lookup(entity, ...)
    Structured row lookup — no FTS. Use for enumerating customers by
    region/CRM stage, competitor profiles, employee lookups, artifact metadata.
    entity values: "customers", "competitors", "employees", "artifacts"
    ** New in v7 ** NOTE: region values in the DB are exactly:
      'ANZ', 'Canada', 'Nordics', 'North America West'

  sql_query(sql)
    ** New in v7, restored from v4/v5 **
    Execute a read-only SELECT. Use when lookup() filters aren't enough —
    exact region match, multi-table joins, or aggregations.
    Tables: customers, artifacts, competitors, employees, implementations,
            scenarios, products. Schema documented inline.

  Decision rules (7):

  Rule 1 — Start with search() for content questions; start with lookup() or
    sql_query() for enumeration (customer lists, competitor profiles,
    employee records).

  Rule 2 (strengthened vs v6) — After search(), if the question asks for
    specific field names, commands, exact numbers, owners, or multi-step
    plans: call read() on the top relevant artifact IDs. Do NOT try to answer
    from snippets when specific values are required.

  Rule 3 — For "is this widespread?" or "which accounts share X?", use
    search(group_by_customer=True) then read() the top artifact IDs to verify
    and enrich each account's situation.

  Rule 4 (new in v7) — For cross-account classification (e.g. "which are
    taxonomy problems vs duplicate problems"), use search(group_by_customer=True)
    with targeted keywords for each category, then read() representative
    artifacts to confirm the classification.

  Rule 5 — For exact action items, owners, or dates: use
    artifact_type="internal_communication" to target Slack threads, which
    contain the most precise action-item lists.

  Rule 6 (new in v7) — Use sql_query() when you need exact column values
    (e.g. region = 'North America West'), aggregations, or joins not covered
    by lookup().

  Rule 7 — If search returns no results, rephrase with different/simpler
    keywords before concluding the data doesn't exist — empty results mean
    vocabulary mismatch.

================================================================================
  TOOLS  (4 total)
================================================================================

  1. search(query, customer_name="", region="", crm_stage="", artifact_type="",
            limit=10, snippet_chars=400, group_by_customer=False)
     FTS over artifact title, summary, and content_text. Returns matching
     artifacts with a content snippet. Set group_by_customer=True to group
     results by customer, each entry includes artifact_count, artifact_ids,
     artifact_titles, and a top_snippet. All filters are partial-match (LIKE).

     ** Fixed in v7 (Bug 1) ** snippet() auxiliary function call corrected:
       snippet(f, 2, ...)  →  snippet(artifacts_fts, 2, ...)
       SQLite FTS5 auxiliary functions require the real table name, not a table
       alias. The alias 'f' was treated as a column name, causing every search()
       call to raise sqlite3.OperationalError: no such column: f. LangGraph's
       ToolNode caught the exception and returned it as a ToolMessage, causing
       the LLM to hallucinate answers rather than erroring visibly.

     ** New in v7 (Bug 2 fix) ** Progressive FTS query fallback via
       _fts_candidates():
       FTS5 AND-mode requires ALL query tokens to appear in a document. Agent
       queries often include paraphrased words absent from documents (e.g.
       "issue" vs "degradation"), silently excluding correct results. The fix
       builds a ladder of progressively shorter AND-mode queries:
         1. Full sanitized query
         2. Drop pure numeric tokens (date fragments: '02', '20', '2026')
         3. Drop trailing tokens one at a time to a 2-token minimum
       search() tries each in order, stopping at the first with ≥ 3 results.
       This preserves AND precision for specific queries while recovering
       gracefully when the agent paraphrases.

       Example: "taxonomy rollout issue" fails AND-mode (BlueHarbor has no
       "issue" in its docs; CedarWind does) → fallback to "taxonomy rollout"
       → BlueHarbor #1, 2, 3.

  2. read(artifact_ids: list[str])
     Fetch full content of one or more artifacts by ID in a single call.
     Returns: artifact_id, artifact_type, title, content_text, customer_name.
     Unchanged from v6.

  3. lookup(entity, name="", region="", crm_stage="", industry="",
            department="", limit=100)
     Structured row lookup. entity values and what 'name' means:
       "customers"   — filters by customer name; also accepts region, crm_stage,
                       industry
       "competitors" — filters by competitor name; returns full profile
       "employees"   — filters by employee name; also accepts department, region
       "artifacts"   — filters by customer name; returns metadata (no body text)
     Unchanged from v6. Note: lookup uses LIKE for region, which fails for
     partial region abbreviations (e.g. "NA West" does not match "North America
     West"). Use sql_query() for exact region matches.

  4. sql_query(sql)  ← RESTORED from v4/v5, removed in v6
     Read-only SELECT against the SQLite database. Rejects non-SELECT statements.
     Full schema documented inline in the tool docstring.

     Why it was needed: In v6, the agent called lookup(region="NA West") for
     q6 (NA West classification). LIKE '%NA West%' does not match the DB value
     'North America West', returning 0 rows. sql_query() allows exact matches
     and joins the lookup tool cannot express.

================================================================================
  BUGS FIXED IN THIS VERSION
================================================================================

  Bug 1 — snippet() alias error (search_agent/tools.py)
    Symptom : sqlite3.OperationalError: no such column: f on every search() call.
              LangGraph ToolNode caught and returned error as ToolMessage;
              the LLM continued but had no real search results.
    Root cause : snippet(f, 2, ...) uses the table alias 'f'; FTS5 auxiliary
              functions require the unaliased virtual table name.
    Fix       : snippet(f, ...)  →  snippet(artifacts_fts, ...)
    Impact    : q1, q3, q5, q6, q13, q17 were all affected. This single bug
              was responsible for the bulk of the v6 regression.

  Bug 2 — FTS AND-mode vocabulary mismatch (search_agent/tools.py)
    Symptom : Correct customer not found when agent's query words differ from
              document vocabulary. q1: "taxonomy rollout issue" matches
              CedarWind (has "issue"), not BlueHarbor (says "degradation").
    Root cause : FTS5 requires ALL tokens to match (AND semantics). One
              unexpected word eliminates correct documents entirely.
    Fix       : _fts_candidates() progressive fallback — tries full query,
              then drops numeric tokens, then drops trailing tokens one at a
              time until ≥ 3 results or only 2 tokens remain.
    Impact    : q1, q3, q5 — all require the agent to discover a customer by
              topic rather than name, making them sensitive to paraphrasing.

  Bug 3 — Region filter mismatch for NA West (v6 design gap)
    Symptom : q6 agent called lookup(region="NA West") → 0 rows returned.
              DB stores region as 'North America West'; LIKE '%NA West%' has
              no substring match.
    Fix       : Restored sql_query() tool + added exact region values to both
              the sql_query docstring and the system prompt lookup NOTE.
    Impact    : q6 went from 0.00 to 0.80.
